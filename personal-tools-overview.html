Repo-Strategie
Wie werden drei separate Repos zu einem harmonisierten System?
⚠
Aktueller Stand: Drei separate Git-Repos (haushaltskasse, agentic_depot_1, work-journal). Das Gespräch mit Claude Web UI schlägt personal-tools/ als Monorepo vor. Das ist eine echte Entscheidung.
✅ Option A — Neues Monorepo (empfohlen)
Vorgehen:
  • Neues Repo personal-tools/ anlegen
  • Code aus den drei Repos rüberkopieren (wenig Code vorhanden)
  • Alte Repos archivieren / als Read-only lassen
  • shared/ einmalig aufbauen

Vorteile:
  • Sauberer Start ohne altes Gepäck
  • Shared-Module sofort nutzbar
  • Ein .env, eine requirements.txt
  • Claude Code hat alles in einem Kontext

Nachteil:
  • Git-History der alten Repos geht verloren (bei wenig Code kein Problem)
Option B — Separate Repos + lokales Package
Vorgehen:
  • personal-tools-shared/ als eigenes Repo
  • Jedes Projekt importiert via pip install -e ../shared
  • Drei Repos bleiben eigenständig

Vorteile:
  • Keine Migration nötig
  • Git-History bleibt erhalten
  • Projekte unabhängiger

Nachteile:
  • Mehr Verwaltungsaufwand
  • Drei separate .env Dateien
  • Risiko der Divergenz höher
Empfohlene Schritte (Option A)
  • 1Neues Repo personal-tools/ anlegen (lokal, dann GitHub)
  • 2shared/ Module zuerst aufbauen: azure_client, llm_client, config, csv_parser
  • 3Work Journal als erstes Projekt einbauen (einfachste Domäne, Schema bereits vorhanden)
  • 4Depot Storage-Layer in Python neu schreiben, dann Workflows aufbauen
  • 5Haushaltskasse als letztes (komplexeste Geschäftslogik)
  • 6Alte Repos auf "archived" setzen

🔧 Shared — Module
Vier gemeinsame Module · Niemals projektspezifisch anpassen
shared/azure_client.py
Generischer Azure Table Storage Wrapper. Methoden: upsert, query, list, delete. Tabellen anlegen wenn nicht vorhanden (idempotent). Alle Projekte nutzen diesen, nie das SDK direkt. Zugriff immer mit Tabellen-Prefix (hh_, depot_, wj_).
shared/llm_client.py
Claude API Wrapper mit Consent-Gate. Modell-Routing: Haiku für Klassifikation/Mini-Tasks, Sonnet für Analyse/Erklärungen. Zeigt vor jedem API-Aufruf was gesendet wird. Kein direkter Anthropic SDK Aufruf außerhalb dieses Moduls.
shared/csv_parser.py
Pandas-basierter CSV/Excel-Parser. Unterstützt DKB-CSV-Format, comdirect-Export, generische CSV. Encoding-Erkennung, Datumsformat-Normalisierung, Duplikat-Erkennung via Hash.
shared/config.py
Zentrale Konfiguration. Lädt .env via python-dotenv. Stellt alle Keys und Einstellungen als typisierte Werte bereit. Wirft sprechende Fehler wenn ein Required-Key fehlt.
Verwendung in Projekten
# In jedem Projekt-Modul:
from shared.azure_client import AzureClient
from shared.llm_client import LLMClient, ModelChoice
from shared.config import config

client = AzureClient(prefix="hh_")  # oder "depot_", "wj_"
llm = LLMClient()

# Consent-Gate automatisch aktiv
result = await llm.analyze(
    prompt="Erkläre diese Transaktionen...",
    model=ModelChoice.SONNET,
    context=snapshot_json
)
🔧 Shared — Entwicklungsregeln
Verbindliche Regeln für alle drei Projekte
1
shared/ bleibt generischNiemals projektspezifischen Code in shared/ — Erweiterungen ja, HH/Depot/WJ-Logik nein.
2
Jedes Projekt ist unabhängig startbarKein Projekt darf ein anderes als Voraussetzung haben. cd haushaltskasse && uvicorn api.router:app muss funktionieren.
3
Azure immer über azure_client.pyNiemals azure-data-tables SDK direkt aufrufen. Nur über shared/azure_client.py.
4
KI immer über llm_client.py mit Consent-GateKein direkter Anthropic SDK Aufruf. Consent-Gate ist Pflicht — nie deaktivieren.
5
Secrets nur in .env, niemals in CodeNiemals hardcoded, niemals in Git. .env ist in .gitignore. .env.example ist immer aktuell zu halten.
6
Tabellennamen immer mit Prefixhh_, depot_, wj_ — niemals ohne Prefix eine Tabelle ansprechen. azure_client.py erzwingt das.
