## 2. Datenmodell — Drei Ebenen

```
Konto              Kategorie (= Nebenbuch, 1:1)        Unterkategorie
(real, Geld liegt  (Ausgabeart + Rücklagen-Topf)       (nur Auswertung)
 physisch hier)
─────────────────  ──────────────────────────────      ────────────────────────
Girokonto          Auto                                 Tanken · KFZ-Steuer · ADAC
Kreditkarte        Sport                                Verein · Ausrüstung · JUDO Club
Tagesgeld          Konsum                               Elektronik · Kleidung · Bücher
Depot              Lebensmittel, Urlaub, Versicherung…  …
```

- **Konto** trägt den **Realsaldo** (echtes Geld).
- **Kategorie = Nebenbuch (1:1)** trägt die **Rücklage**. Nur hier gibt es Rücklagen-Töpfe.
- **Unterkategorie** ist eine **reine Auswertungsdimension** innerhalb einer Kategorie —
  **keine eigene Rücklage, kein eigener Saldo**. Sie dient dem Drill-Down.
- Der **Empfänger** wird auf *jeder* Buchung gespeichert. Drill-Down bis zum einzelnen
  Empfänger ist damit immer möglich, auch ohne gesetzte Unterkategorie. Die Unterkategorie
  ist die optionale **Gruppierungsschicht darüber** (fasst z. B. Agip/bft/Shell zu „Tanken").
  Wo keine Gruppierung sinnvoll ist, *ist* der Empfänger praktisch die Unterkategorie.

### Zwei Salden (Rücklagen-Prinzip)
- **Realsaldo** = Summe aller realen Buchungen über alle Konten.
- **Verfügbarer Saldo** = Realsaldo − Summe aller Rücklagen.
- Ziel: der *verfügbare* Saldo bleibt glatt; die Rücklage fängt klumpige Jahres-/Quartalszahlungen ab.

### Drei Buchungsarten
1. **Realbuchung** — Konto + Kategorie, ändert Realsaldo.
2. **Rücklagenbuchung** — virtuell: Zuführung (+), Verzehr bei realer Ausgabe (−), Korrektur (±).
3. **Umbuchung** — Transfer zwischen zwei Konten, netto null, **keine** Kategorie.
   Kreditkarten-Ausgleich (Giro → Kreditkarte) ist eine Umbuchung und darf **nicht**
   als neue Ausgabe zählen (sonst Doppelzählung — die Einzelkäufe sind die echten Ausgaben).

## 3. Unterkategorien — Regeln (verbindlich)

1. **Nur Auswertung.** Keine Rücklage auf Unterkategorie-Ebene. Das Zwei-Salden-Modell bleibt einfach.
2. **Wachsend, KI-gestützt.** Bei unbekannten Empfängern schlägt die KI eine passende oder
   **neue** Unterkategorie vor. Der Nutzer kann jeden Vorschlag:
   - **annehmen**,
   - **ablehnen** (Regel-Status `abgelehnt`, wird nie wieder vorgeschlagen),
   - **umdefinieren** (anderer Name / andere Kategorie),
   - **eigene ergänzen** (manuell anlegen, unabhängig von KI).
3. **Rückwirkend.** Weil Unterkategorien reine Auswertung sind, gilt jede Änderung/Neudefinition
   **über die gesamte Historie**: Ändern einer Mapping-Regel oder Umbenennen einer Unterkategorie
   löst ein Reapply über alle passenden Buchungen aus. **Ausnahme:** manuell fixierte Buchungen
   (`unterkat_pinned = 1`) bleiben unangetastet.

---

## 4. KI-gestützte Zuordnung (Pipeline)

Für jede importierte Buchung, in dieser Reihenfolge — günstig zuerst, KI nur im Notfall:

1. **Regel-Lookup (kostenlos).** `mapping_regeln` durchsuchen (Empfänger / Verwendungszweck /
   IBAN / SEPA-Ref, Teilstring, case-insensitive, Status `aktiv`). Treffer → Kategorie
   (+ Unterkategorie) direkt setzen. Nach kurzer Einlernzeit die Mehrheit der Buchungen.
2. **KI-Fallback (Haiku, Klassifikation).** Nur bei unbekanntem Muster. Claude schlägt
   `{Kategorie, Unterkategorie}` vor und darf eine neue Unterkategorie vorschlagen.
   **Consent-Gate** (`shared/llm_client`) vor jedem Aufruf; es wird nur ein minimaler Snapshot gesendet.
3. **Bestätigung → Lernen.** Die Nutzer-Entscheidung wird als `mapping_regeln`-Eintrag
   (`quelle = manuell` bzw. bestätigte `ki`) zurückgeschrieben → beim nächsten Mal wieder
   deterministisch, kein API-Call mehr.

Damit ist „Erkennung und Zuordnung automatisiert und KI-unterstützt" erfüllt: Regeln wo bekannt,
KI wo neu, jede Bestätigung trainiert die Regeln.

## 5. Storage-Schema (SQLite)

Umgesetzt in `haushaltskasse/storage/schema.sql`. Tabellen:

| Tabelle | Zweck |
|---|---|
| `konten` | reale Konten (giro/kreditkarte/tagesgeld/depot) |
| `kategorien` | Nebenbücher 1:1 inkl. `monatliche_ruecklage_cent` |
| `unterkategorien` | Auswertungsdimension je Kategorie (quelle ki/manuell) |
| `buchungen` | alle Bewegungen; `buchungsart` real/ruecklage/umbuchung; `*_pinned` schützt manuelle Zuordnung vor Reapply; `import_hash` dedupliziert |
| `mapping_regeln` | lernende Muster→Kategorie/Unterkategorie; `status` aktiv/vorschlag/abgelehnt |
| `import_dateien` | Import-Audit |

Konventionen: Beträge als **Cent (INTEGER)**, Datum als **ISO-8601 Text**. Kein `hh_`-Prefix
(eigene DB-Datei; die Prefix-Regel galt der geteilten Azure-Namespace).
